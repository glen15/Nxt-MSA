## 목표
강사의 공장 서버에 접속하여 3개 공장의 API 차이를 직접 확인하고,
**강결합 vs 느슨한 결합**의 차이를 이해한다.

## 사전 준비
- Lab 01 완료 (NxtCar 메인 앱 정상 동작)
- 강사가 안내한 **공장 서버 IP** (예: `3.95.xx.xx`)

---

## Step 1: 공장 허브 대시보드 접속 (10분)

### 1-1. 공장 허브 열기

브라우저에서 강사가 안내한 공장 IP의 포트 3000에 접속합니다:

```
http://<공장IP>:3000
```

**공장 허브 대시보드**가 열립니다. 상단 탭에서 전체 현황과 개별 공장을 확인할 수 있습니다:

| 탭 | 내용 |
|---|---|
| 📊 전체 | 3개 공장 통합 현황 |
| 🔧 엔진 공장 | 엔진 공장 상세 |
| 🛞 타이어 공장 | 타이어 공장 상세 |
| 🔋 배터리 공장 | 배터리 공장 상세 |
| 📋 API 문서 | 3개 공장의 API 명세 |
| 🧩 아키텍처 | Lambda 어댑터 패턴 설명 |

> 아직 아무도 생산 요청을 보내지 않았으므로 "생산 요청 대기 중..."이 표시됩니다.

### 1-2. 개별 공장 대시보드

각 공장도 자체 대시보드를 가지고 있습니다. 직접 접속해보세요:

| 공장 | URL | 색상 |
|------|-----|------|
| 🔧 엔진 공장 | `http://<공장IP>:3001` | 빨강 |
| 🛞 타이어 공장 | `http://<공장IP>:3002` | 파랑 |
| 🔋 배터리 공장 | `http://<공장IP>:3003` | 초록 |

각 공장은 독립된 서버입니다. 포트가 다르고, 담당 부품이 다릅니다.

## Step 2: API 문서 확인 — "같은 일, 다른 방법" (15분)

공장 허브의 **📋 API 문서** 탭을 열어보세요.

3개 공장이 하는 일은 같습니다: **부품을 생산**합니다.
그런데 API가 **전부 다릅니다.**

### 생산 요청

| | 🔧 엔진 공장 | 🛞 타이어 공장 | 🔋 배터리 공장 |
|---|---|---|---|
| 엔드포인트 | POST `/api/produce` | POST `/api/manufacture` | POST `/api/orders` |
| 응답 코드 | **202** Accepted | **202** Accepted | **200** OK |
| 작업 ID 필드 | `jobId` | `manufacturingId` | `orderNumber` |

같은 "생산 요청"인데 경로도, 응답 코드도, 필드 이름도 전부 다릅니다.

### 상태 조회

| | 🔧 엔진 공장 | 🛞 타이어 공장 | 🔋 배터리 공장 |
|---|---|---|---|
| 엔드포인트 | GET `/api/status/:id` | GET `/api/jobs/:id` | POST `/api/orders/status` |
| HTTP 메서드 | **GET** | **GET** | **POST** |

배터리 공장은 상태 조회조차 **POST**입니다.

### 헬스 체크

| | 🔧 엔진 공장 | 🛞 타이어 공장 | 🔋 배터리 공장 |
|---|---|---|---|
| 엔드포인트 | `/api/health` | `/api/health` | `/api/ping` |
| 응답 | JSON | JSON | 텍스트 `pong` |

> 🤔 **현실에서도 이렇습니다.**
> 외부 업체의 API는 우리가 통제할 수 없습니다.
> 같은 "생산 요청"이라도 업체마다 경로, 필드명, 응답 형식이 전부 다릅니다.
> 이 상황에서 우리 서버는 어떻게 3개 공장과 통신해야 할까요?

## Step 3: 강결합 — "직접 호출하면 어떻게 되는가" (15분)

가장 단순한 방법: NxtCar 서버가 공장 API를 **직접 호출**합니다.

### 강결합 코드 (의사 코드)

```javascript
// ❌ 강결합: NxtCar가 공장을 직접 호출
async function orderParts(partId, quantity, category) {

  if (category === 'engine') {
    // 엔진 공장: POST /api/produce, jobId로 응답
    await fetch('http://엔진공장:3001/api/produce', {
      body: { purchaseOrderId, partId, quantity }
    });

  } else if (category === 'tire') {
    // 타이어 공장: POST /api/manufacture, manufacturingId로 응답
    await fetch('http://타이어공장:3002/api/manufacture', {
      body: { purchaseOrderId, partId, quantity }
    });

  } else if (category === 'battery') {
    // 배터리 공장: POST /api/orders, orderNumber로 응답
    await fetch('http://배터리공장:3003/api/orders', {
      body: { purchaseOrderId, partId, quantity }
    });
  }
}
```

동작은 합니다. 하지만 **문제가 4가지** 있습니다.

### 문제 1: 장애 전파

```
엔진 공장 서버 다운
     ↓
NxtCar가 직접 호출 → 타임아웃 30초 대기 → 에러
     ↓
고객의 주문 API도 실패
     ↓
"차량 주문이 안 돼요" ← 엔진 공장 때문에 전체 서비스 장애
```

공장 하나의 장애가 **우리 서비스까지 전파**됩니다.
엔진 공장이 죽었을 뿐인데, 타이어만 필요한 전기차 주문까지 영향을 받을 수 있습니다.

### 문제 2: 공장 추가 = NxtCar 수정

"브레이크 공장이 추가되었습니다."

```javascript
// NxtCar 코드를 수정해야 한다
} else if (category === 'brake') {
    await fetch('http://브레이크공장:3004/api/request-production', {
      body: { ... }  // 또 다른 형식
    });
}
```

공장이 추가될 때마다 **NxtCar 코드를 수정하고 재배포**해야 합니다.
NxtCar와 공장이 1:1로 묶여 있기 때문입니다.

### 문제 3: 동기 블로킹

```
고객 주문 → NxtCar가 공장 호출 → 3~8초 대기... → 응답 → 고객에게 반환
```

공장의 생산은 수 초~수 분이 걸립니다.
직접 호출하면 NxtCar가 **공장 응답을 기다리는 동안** 고객 요청도 멈춥니다.

### 문제 4: 메시지 유실

```
NxtCar → 공장 호출 → 네트워크 에러 → 발주 사라짐
```

재시도 로직을 직접 구현하지 않으면, 네트워크 문제로 발주가 **조용히 사라집니다**.
"분명 발주했는데 공장에서는 받은 적이 없다"는 상황이 발생합니다.

### 강결합 정리

```
NxtCar ──직접호출──→ 엔진 공장    ← NxtCar가 URL, API 형식, 인증을 전부 알아야 함
       ──직접호출──→ 타이어 공장   ← 하나라도 죽으면 NxtCar도 영향
       ──직접호출──→ 배터리 공장   ← 공장 추가/변경 시 NxtCar 수정 필요
```

## Step 4: 느슨한 결합 — "메시지로 끊어내기" (15분)

해결: NxtCar와 공장 사이에 **메시지 시스템**을 넣습니다.

### 느슨한 결합 코드 (실제 코드)

```javascript
// ✅ 느슨한 결합: NxtCar는 SNS에 메시지만 발행
async function orderParts(partId, quantity, category) {
  await sns.publish({
    TopicArn: ORDERING_TOPIC,
    Message: JSON.stringify({
      purchaseOrderId, partId, quantity, category
    }),
    MessageAttributes: {
      category: { DataType: 'String', StringValue: category }
    }
  });
  // 끝. 누가 받든, 어떻게 처리하든 NxtCar는 모른다.
}
```

**if/else가 없습니다.** NxtCar는 공장의 존재 자체를 모릅니다.
"엔진 20개 필요"라는 **메시지를 던질 뿐**입니다.

### 메시지 흐름

```
NxtCar ──메시지──→ SNS (ordering 토픽)
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     SQS engine   SQS tire    SQS battery    ← SNS 필터가 자동 분배
          │            │            │
          └────────────┼────────────┘
                       ▼
                  Lambda (통합)              ← category 보고 공장 호출
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     엔진 공장    타이어 공장   배터리 공장
```

### 문제 해결 비교

| 문제 | 강결합 | 느슨한 결합 |
|------|--------|-----------|
| 공장 다운 | NxtCar도 에러 | SQS에 메시지 쌓임, 복구 후 자동 처리 |
| 공장 추가 | NxtCar 코드 수정 + 재배포 | SQS 구독 추가만. NxtCar 변경 없음 |
| 응답 대기 | 공장 응답까지 블로킹 | SNS 발행 즉시 반환 (수 ms) |
| 네트워크 에러 | 발주 유실 | SQS가 3회 재시도 + DLQ 보관 |

### 각 구성요소의 역할

| 구성요소 | 역할 | 비유 |
|---------|------|------|
| **SNS** | 메시지 발행 + 필터링 | 우체국 — 주소(category) 보고 분류 |
| **SQS** | 메시지 저장 + 재시도 | 우편함 — 수신자가 없어도 보관 |
| **Lambda** | 메시지 → API 호출 변환 | 번역기 — SQS 형식을 공장 API로 변환 |

> 💡 **핵심**: NxtCar는 공장의 URL도, API 형식도, 몇 개인지도 모릅니다.
> "엔진이 필요하다"는 사실만 SNS에 알리면, 나머지는 메시지 시스템이 처리합니다.

## Step 5: 어댑터 패턴 — Lambda의 역할 (5분)

3개 공장의 API가 전부 다르다는 문제는 여전히 남아 있습니다.
그 차이를 **Lambda가 흡수**합니다.

```
  통일된 SQS 메시지            Lambda가 변환           공장별 API
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│ partId           │     │ if engine:       │     │ POST /api/produce│
│ quantity         │ ──→ │   /api/produce   │ ──→ │ jobId 응답       │
│ category         │     │ if tire:         │     │                  │
│ purchaseOrderId  │     │   /api/manufacture│──→ │ POST /api/manufacture│
│                  │     │ if battery:      │     │ manufacturingId 응답│
│                  │     │   /api/orders    │ ──→ │ POST /api/orders  │
└─────────────────┘     └──────────────────┘     └─────────────────┘
```

- **NxtCar** → 통일된 메시지 형식으로 발행 (변환 불필요)
- **Lambda** → 메시지를 읽고, category에 맞는 공장 API로 변환하여 호출
- **공장** → 자기 방식대로 처리 (변경 불필요)

> 이 Lambda가 **어댑터(Adapter)**입니다.
> 110V 콘센트에 220V 기기를 꽂을 때 변압기를 쓰는 것과 같습니다.

## 교육 포인트 정리

### 이 Lab에서 배운 것
1. **API 다양성**: 같은 "생산 요청"이지만 공장마다 엔드포인트, 응답, 필드가 다르다
2. **강결합의 위험**: 직접 호출 시 장애 전파, 코드 수정 비용, 동기 블로킹, 메시지 유실
3. **느슨한 결합**: 메시지 시스템(SNS/SQS)으로 서비스 간 의존성을 끊는다
4. **어댑터 패턴**: Lambda가 통일된 메시지를 공장별 API로 변환하는 번역기 역할
