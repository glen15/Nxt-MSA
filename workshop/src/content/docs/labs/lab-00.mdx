## 시나리오

NxtCar는 차량 조립 공장입니다.
세단, 전기차, SUV를 생산하는데, 차량마다 필요한 부품이 다릅니다.

| 차량 | 타입 | 엔진 | 타이어 | 배터리 |
|------|------|:----:|:------:|:------:|
| 세단 | 내연기관 | 1 | 4 | - |
| 전기차 | EV | - | 4 | 2 |
| SUV | 하이브리드 | 1 | 4 | 1 |

차량을 주문하면 부품 재고가 차감됩니다.
재고가 임계치 이하로 떨어지면 **부품 공장에 자동 발주**해야 합니다.

문제는 — 부품 공장이 3곳이고, API가 전부 다릅니다.

## 핵심 개념

### 1. 마이크로서비스 아키텍처 (MSA)

모놀리식과 달리 각 서비스가 독립적으로 배포·확장됩니다.

| 모놀리식 | MSA |
|---------|-----|
| 하나의 서버에 모든 기능 | 기능별로 서비스를 분리 |
| 하나가 죽으면 전부 다운 | 공장 하나가 죽어도 주문은 계속 됨 |
| 배포할 때 전체 재시작 | 해당 서비스만 배포 |

이 실습에서는:
- **NxtCar API 서버** = 주문 + 재고 관리
- **엔진/타이어/배터리 공장** = 각각 독립된 서비스
- **Lambda 어댑터** = 서비스 간 연결

### 2. 느슨한 결합 (Loose Coupling)

NxtCar 서버는 공장의 URL, API 스펙, 인증 방식을 **모릅니다**.

SNS에 "엔진 100개 필요"라는 메시지를 보낼 뿐, 누가 받아서 어떻게 처리하는지는 관심 없습니다.

**이점:**
- 공장 추가 → SQS 구독 + Lambda만 추가. NxtCar 코드 변경 없음
- 공장 API 변경 → 해당 Lambda만 수정
- 공장 다운 → SQS에 메시지 적체, 복구 후 자동 처리. NxtCar는 정상

**만약 타이트하게 결합되어 있다면:**
- NxtCar가 공장 3개의 URL과 API를 전부 알아야 함
- 공장 하나 추가할 때마다 NxtCar 코드 수정 + 재배포
- 공장이 다운되면 NxtCar도 에러

### 3. Pub/Sub + Fan-out

하나의 이벤트(재고 부족)를 여러 구독자에게 동시에 전달합니다.

```
SNS 발주 토픽에 메시지 발행
  → MessageAttribute: { category: "engine" }

SNS가 필터링해서 해당 큐에만 전달:
  engine-queue  ← category = "engine" 일 때만
  tire-queue    ← category = "tire" 일 때만
  battery-queue ← category = "battery" 일 때만
```

SUV를 주문하면 엔진+타이어+배터리 3개 부품이 필요합니다.
3개 부품 모두 임계치 이하라면, **3개 메시지가 동시에 3개 큐로** 팬아웃됩니다.

### 4. 어댑터 패턴 (Lambda)

3개 공장의 API가 **의도적으로 전부 다릅니다:**

| | 엔진 공장 | 타이어 공장 | 배터리 공장 |
|---|---|---|---|
| 생산 요청 | POST `/api/produce` | POST `/api/manufacture` | POST `/api/orders` |
| 응답 코드 | 202 | 202 | **200** |
| 상태 조회 | GET `/api/status/:id` | GET `/api/jobs/:id` | **POST** `/api/orders/status` |
| 헬스 체크 | `/api/health` | `/api/health` | **`/api/ping`** |

Lambda가 **통일된 SQS 메시지 → 공장별 API 호출**로 변환합니다.
이것이 어댑터 패턴입니다. 각 Lambda가 번역기 역할을 합니다.

### 5. 최종 일관성 (Eventual Consistency)

주문 → 재고 차감 → 발주 → 생산 → 입고가 **비동기**로 진행됩니다.

```
시점 1: 주문 접수       → 재고 차감 (즉시)
시점 2: 발주 메시지 발행 → SNS/SQS (수 밀리초)
시점 3: Lambda → 공장   → 생산 시작 (수 초)
시점 4: 생산 완료       → 입고 메시지 (수 초~수 분)
시점 5: 재고 충전       → DynamoDB 업데이트 (수 밀리초)
```

모든 것이 즉시 반영되지 않지만, **결국에는(eventually)** 일관된 상태가 됩니다.
이 사이에 발생하는 "아직 입고 안 된" 상태도 정상입니다.

### 6. 장애 복구 (DLQ + Redrive)

Lambda가 공장 호출에 실패하면?

```
SQS → Lambda → 공장 API (실패!)
  ↓ 3회 재시도 후
DLQ (Dead Letter Queue) 로 이동
  ↓ 장애 복구 후
Redrive → 다시 SQS → Lambda → 공장 API (성공)
```

메시지가 유실되지 않습니다. DLQ에서 대기하다가 Redrive로 재처리됩니다.

## 사용하는 AWS 서비스

| 서비스 | 역할 | 이 실습에서 |
|--------|------|-----------|
| **S3** | 정적 웹 호스팅 | NxtCar 프론트엔드 |
| **EC2 / Cloud9** | 컴퓨트 | NxtCar API 서버 |
| **DynamoDB** | NoSQL 데이터베이스 | 부품 재고, 주문, 발주 |
| **SNS** | Pub/Sub 메시징 | 발주 알림, 입고 알림 |
| **SQS** | 메시지 큐 | 부품별 발주 큐, 입고 큐 |
| **Lambda** | 서버리스 함수 | 어댑터 (SQS → 공장 API) |
